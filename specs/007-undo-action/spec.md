# Feature Specification: Undo Action and Bottom Controls

**Feature Branch**: `007-undo-action`
**Created**: 2026-03-04
**Status**: Draft
**Input**: User description: "create an undo button, the undo will remove a placed queen and all the check markers and will remove a single placed mark, or multiple if they were set in multiple. put the buttons at the bottom of the puzzle as well as put the reset button at the bottom"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Undo a Queen Placement (Priority: P1)

A player places a queen on the board and immediately realizes it was the wrong cell. They press the Undo button at the bottom of the puzzle and the queen is removed along with any auto-marks that were created as a result of that queen placement.

**Why this priority**: Undoing a queen placement is the core undo action — queens are the primary game piece, and each one can trigger many automatic marks. This is the highest-value undo scenario.

**Independent Test**: Place a queen on any cell; verify auto-marks appear on invalid cells. Press Undo; verify the queen is removed and all auto-marks associated with that queen disappear. The board should be exactly as it was before the queen was placed.

**Acceptance Scenarios**:

1. **Given** the board has no pieces, **When** the user places a queen, **Then** the Undo button becomes enabled.
2. **Given** a queen has been placed (with or without auto-marks), **When** the user presses Undo, **Then** the queen is removed and all auto-marks generated by that queen are removed simultaneously.
3. **Given** the user placed two queens, **When** the user presses Undo twice, **Then** the board is empty after both undos.
4. **Given** the board is empty (no history), **When** the user views the Undo button, **Then** the Undo button is visually disabled and non-interactive.
5. **Given** a queen was placed and then removed by the user (cycled back to empty), **When** the user presses Undo, **Then** the undo applies to the previous recorded action, not the removed queen.

---

### User Story 2 - Undo a Manual Mark Placement (Priority: P2)

A player places one or more X marks on the board (either by clicking a single cell or by dragging across multiple cells in one gesture). They press Undo and all marks placed in that single interaction are removed together.

**Why this priority**: Manual marks are the second type of user action. Batch undo (single drag = single undo step) is necessary to prevent users from having to undo cell-by-cell after a drag.

**Independent Test**: Drag across 4 cells to place 4 X marks; press Undo once; verify all 4 marks disappear together. Then click a single cell to place one X mark; press Undo once; verify that single mark disappears.

**Acceptance Scenarios**:

1. **Given** the user clicks a single empty cell to place an X mark, **When** the user presses Undo, **Then** that single X mark is removed.
2. **Given** the user drags across 3 cells placing X marks in one drag gesture, **When** the user presses Undo once, **Then** all 3 X marks placed in that gesture are removed together.
3. **Given** the user placed a manual mark, then placed a queen, **When** the user presses Undo once, **Then** only the queen (and its auto-marks) is removed; the manual mark remains.

---

### User Story 3 - Bottom-Positioned Controls (Priority: P3)

The Undo button and Reset button both appear at the bottom of the puzzle board, grouped together. The Reset button is moved from its current location in the header to this bottom area.

**Why this priority**: Layout change that improves control accessibility and groups related actions. Lower priority than functional undo, but required as specified.

**Independent Test**: Open any puzzle; verify no Reset button appears in the header; verify both Undo and Reset buttons appear below the puzzle board.

**Acceptance Scenarios**:

1. **Given** the puzzle page loads, **When** the user views the page, **Then** the Reset button is no longer in the header area.
2. **Given** the puzzle page loads, **When** the user views the area below the board, **Then** both an Undo button and a Reset button are visible.
3. **Given** both buttons are visible, **When** the user views them, **Then** they are clearly labeled and visually distinct from the board.
4. **Given** a puzzle is solved, **When** the user views the bottom controls, **Then** both buttons are disabled (puzzle is complete, no actions to undo or reset).

---

### Edge Cases

- What happens when there is no history (no moves made)? The Undo button is disabled and pressing it has no effect.
- What happens if the user removes a queen manually (by clicking a queen cell to cycle it back to empty)? This removal itself is recorded as an undo step — pressing Undo re-places the queen.
- What if the user undoes past a queen that was placed via the X-marked → Queen cycle? The undo reverses to the state before the queen was placed (which means the cell goes back to X-marked, since that was its previous state).
- What happens when the puzzle is solved? Undo and Reset buttons are both disabled — no further changes to the board are allowed.
- What if auto-mark is disabled? Undoing a queen placement still removes the queen; no auto-marks were generated, so none are removed.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The game session MUST maintain an ordered history of user actions that can be stepped back through one action at a time.
- **FR-002**: Each entry in the action history MUST capture the complete state delta needed to reverse that action (queen coordinates, associated auto-marks, or manual mark keys placed in a single interaction).
- **FR-003**: Placing a queen MUST record an action containing: the queen's coordinates and the set of auto-mark keys generated by that queen (if any).
- **FR-004**: Placing one or more manual X marks in a single interaction (single click or single drag gesture) MUST record a single action containing all mark keys placed in that interaction.
- **FR-005**: Pressing Undo MUST reverse the most recent recorded action and remove it from the history.
- **FR-006**: Pressing Undo when a queen was the last action MUST remove the queen and all auto-marks associated with that queen simultaneously.
- **FR-007**: Pressing Undo when a manual mark batch was the last action MUST remove all marks in that batch simultaneously.
- **FR-008**: The Undo button MUST be visually disabled when the action history is empty.
- **FR-009**: The action history MUST be cleared when a new stage is loaded or when the user presses Reset.
- **FR-010**: An Undo button MUST appear in the puzzle controls area below the board.
- **FR-011**: The Reset button MUST be relocated from the header to the puzzle controls area below the board.
- **FR-012**: Both the Undo and Reset buttons MUST be disabled when the puzzle is solved.
- **FR-013**: The header MUST no longer contain the Reset button after this change.

### Key Entities

- **ActionHistory**: An ordered stack of reversible moves in the current game session. Cleared on stage load or reset. Each entry is one of:
  - *QueenPlacement*: records the placed queen's coordinates and the auto-mark keys added as a result.
  - *ManualMarkBatch*: records the set of mark keys added in a single click or drag gesture.
- **BottomControls**: The UI area beneath the puzzle board that houses the Undo button and the Reset button.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Pressing Undo once after placing a queen restores the board to its exact pre-placement state — verified across all puzzle sizes.
- **SC-002**: Pressing Undo once after a drag-mark gesture removes all marks placed in that gesture — verified with drags of 1, 3, and 5+ cells.
- **SC-003**: The action history correctly tracks up to the full number of moves in a puzzle session; every single Undo press steps back exactly one user action.
- **SC-004**: The Reset button is absent from the header in 100% of puzzle page renders.
- **SC-005**: Both Undo and Reset buttons are visible and reachable below the board without scrolling on a standard mobile screen (375 px wide).
- **SC-006**: The Undo button is in a visually disabled state whenever there are no recorded actions to undo.

## Assumptions

- The action history is ephemeral — it is not persisted to localStorage and is reset on page reload, stage load, or reset.
- Removing a queen by clicking it (cycling queen → empty) is itself an undoable action; pressing Undo re-places the queen.
- When a cell is cycled from X-marked to queen (via the click cycle), this is recorded as a single queen-placement action; undoing it returns the cell to X-marked.
- Redo (re-applying undone actions) is out of scope for this feature.
- The history stack has no enforced maximum size for this iteration; the entire session history is kept.
- The hint text and aria-live region currently positioned below the board remain; the new bottom controls are added after the board and before (or after) the hint text, maintaining a logical reading order.
